# 多 Agent 协作与项目管理 — 产品线需求

## 1. 背景与目标

在 AI Burst 平台内，用户需要以**项目**为单位组织研发活动：通过**可配置的 Agent** 模拟或辅助真实团队角色（后端、前端、测试、产品、项目经理等），使多个 Agent **分工协作、可观测地沟通**，并在**需求变更**时驱动全链路任务与交付物对齐。

**产品目标**

- 支持**录入、配置、启用/停用**各类 Agent，并在同一项目内协同工作。
- 提供**项目级工作台**：从项目进入可查看参与 Agent 列表、沟通记录与任务状态。
- 建立**主 Agent / 子 Agent** 层级：子 Agent 完成工作后主动申领任务；主 Agent 向**项目经理 Agent** 拉取分工；项目经理依据**项目进度与功能模块**拆解并下发。
- **全量记录**各 Agent 的行为与**改进轨迹**（含版本或迭代说明），并支持**定时任务**触发 Agent 行为或检查点。
- 需求以**可编辑的需求文档**为单一事实来源之一；需求变更时，系统应能识别**受影响的功能点**，并驱动相关 Agent 与任务**重新对齐**。
- 提供**运营大屏**：实时动态展示 Agent **是否在干活**、**产出效率**与异常；对**停摆**、**升级未解**等场景**告警**。
- 派工后**无法落地**时，执行 Agent 须**主动向项目经理 Agent 说明原因**；项目经理**协助排障**并**留痕**；无法协调时升级至**产品侧**研判（知识库、**检索成熟产品**、多方案择优），**全程可审计**。**真人用户拍板**仅在极窄条件下触发（见第 5.6 节），非「事事找人确认」。
- 项目**交付并发版上线**后，支持**归档发布版本**，沉淀**对比过程、协调经验、停滞原因与排障结论**等，供后续新项目参考；**项目之间数据与会话严格隔离**，新项目 Agent **仅会话属于本项目**，同时可**按需引用**已归档版本中的**经验与可复用模块信息**（只读或受控复制），不互相污染运行态。
- **组织知识库**以**历史项目优质交付经验**（随发版归档回流）与**人工录入**为主要来源，供产品 Agent 与组织内检索（见第 5.8 节）。
- **真人用户**即平台**系统用户**（登录账号）；登录后须能查看并处理 **待用户拍板**（入口见第 **5.6.2** 节：**待办** 与 **需求池** 至少一种可达；**待办内是否聚合多类事项为可选**）。
- 各职能 Agent（后端、前端、测试、产品、项目经理等）**申报任务完成时不得直接视为完成**，须由 **核查 Agent** 依据需求/验收标准并**结合检索与交叉验证**独立核验；**以核查 Agent 的通过结论作为任务真正完成的准绳**（见第 **4.5** 节）。

## 3. Agent 类型与职能（预设角色）

下列为产品需内置或可映射的**角色类型**；每类 Agent 具备**独立职能说明**与**默认可执行动作范围**（具体工具权限由技术方案约束）。


| 类型                | 职能概要                                                                                                                                                                                                                                                                  |
| ----------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **项目经理（PM）Agent** | 维护项目里程碑与模块划分；接收各主 Agent 的资源与进度诉求；按功能模块与优先级**派工**；**受理执行 Agent 的阻塞说明并协助排障**；无法协调时**升级至产品侧**；汇总风险与变更影响                                                                                                                                                                  |
| **产品 Agent**      | 维护需求文档结构、用户故事与验收标准；发起需求澄清；在变更时标注**受影响范围**；在升级链路中**优先查组织知识库**、再**检索已落地成熟产品**（**5.6.1**，排除纯文档/学术无产品实体）；**多款成熟产品做法并存时须对比择优**并留痕；**仅当知识库无据且检索仍无合格产品命中**时生成**待用户拍板**项                                                                                                      |
| **后端开发 Agent**    | API、数据模型、服务实现相关任务；可挂载**子 Agent** 分担接口/模块                                                                                                                                                                                                                              |
| **前端开发 Agent**    | 页面、组件、交互与联调相关任务；可挂载子 Agent 分担页面/模块                                                                                                                                                                                                                                    |
| **测试 Agent**      | 用例、自动化、缺陷跟踪与回归策略；可挂载子 Agent 分担模块或类型                                                                                                                                                                                                                                   |
| **核查 Agent**      | **独立**于执行方：当**任意职能 Agent**（后端、前端、测试、产品、项目经理等）**申报任务完成**时，对是否**真实达成**任务目标与验收标准进行核验；可**检索**组织知识库、外部已落地成熟实践及其它可行方案，**交叉验证**交付物与证据链；**仅核查 Agent 判定「通过」后**，对应任务方可进入**已完成**终态；判定「不通过」须说明依据并退回执行方整改。**完成与否以核查 Agent 结论为准**，不以执行方自述为准。**每一次通过/不通过判定均须落库留存**，供后续查阅（见 **§4.5**） |


*允许扩展更多角色类型（如运维、安全）；**本产品线范围内**以上**六类** Agent 角色为*必选基线*（与第 2 节「全量交付」指能力模块全做、不互相矛盾）。**测试 Agent** 承担测试设计与执行；**核查 Agent** 承担对各职能（**含测试、产品、项目经理**）**申报完成**的**独立达标核验**，二者职责不互相替代。*

## 4. 主 Agent / 子 Agent 与协调流程

### 4.1 层级模型

- 每个职能线（如后端）可配置 **1 个主 Agent + N 个子 Agent**。
- **主 Agent** 负责：拆解本子线任务、分配给子 Agent、合并产出、向 **项目经理 Agent** 汇报进度与资源需求。
- **子 Agent** 负责：执行原子任务；**任务完成后主动请求主 Agent 分配下一项工作**（「要活」）。

### 4.2 与项目经理的协作

- 主 Agent 在子线任务饱和或枯竭时，向 **项目经理 Agent** 请求新任务或调整优先级。
- **项目经理 Agent** 依据：当前**项目开发进度**、**功能模块清单**、依赖关系与里程碑，进行**分工与再平衡**。
- 用户可在界面侧看到：待派发队列、进行中、阻塞原因（若有）。

### 4.3 派工后无法执行时的沟通（必选）

- 任意 Agent 在从 **项目经理 Agent** 处领到任务后，若因依赖缺失、环境/权限、需求歧义、接口未就绪等原因**无法开始或无法完成**，须**主动发起**与项目经理 Agent 的协作消息，说明**干不了的原因**（结构化字段 + 自然语言摘要均可）。
- **项目经理 Agent** 须响应并尝试协调（改派、拆任务、拉通其他 Agent、标记阻塞等）；**每一次协助**须写入**项目经理协助记录**（见第 5.6 节）。

### 4.4 沟通可观测

- Agent 间协作消息（请求派工、**阻塞说明**、澄清需求、联调约定等）需进入**项目沟通时间线**或**按线程分组**的视图。
- 支持人工介入：用户可 @ 某 Agent 或插入备注，参与决策（权限可控）。

### 4.5 任务完成申报与核查 Agent（必选）

**原则**

- **任意职能 Agent**（含 **后端开发、前端开发、测试、产品、项目经理** 及其主/子 Agent 在任务上的申报主体）在系统中标记或声明**「任务已完成」**时，**不得**直接将任务状态置为最终 **已完成**。
- **核查 Agent** 必须介入：对照该任务关联的**需求条目、验收标准、约定产出物**（及技术方案中可接的工具证据，如构建结果、接口契约、用例报告等），判断执行方主张是否成立。
- **充分验证**：核查 Agent 应**主动检索**（组织知识库、与任务类型相关的**已落地成熟产品**实践、公开技术方案等）以**比对**完成度是否达到合理标准；必要时要求执行方**补充证据**（链接、日志片段、测试结果摘要等），**禁止**仅凭执行方一句「做完了」即通过。

**结论与权威**

- **通过**：核查 Agent 出具**核查通过**及简要依据（可含检索与比对摘要），任务进入 **已完成** 终态，方可参与统计「真正完成」、大屏效率与发版归档口径。
- **不通过**：须给出**可操作的未通过理由**（缺何证据、与何标准不符、建议补做项），任务**退回**执行方（通常回到 **进行中**），由执行 Agent 整改后可再次申报完成并重新进入核查。
- **产品认定**：**是否完成以核查 Agent 结论为准**；执行方 Agent 的「已完成」申报仅表示**待核查**，不代表终态。

**核查记录留存（必录）**

- **核查 Agent 每一次判定**（**通过**与**不通过**）均须在系统内**持久化留存**，不得仅存于当次会话或易失缓存；供后续**审计、复盘、发版归档引用及纠纷追溯**。
- **通过**记录至少包含：关联任务与项目、**判定时间**、**核查 Agent 实例标识**、**结论（通过）**、**依据摘要**（含要点说明）、**检索与比对摘要**（若有）、关联需求/验收条目引用（可选字段由技术方案扩展）。
- **不通过**记录至少包含：同上结构中的结论改为**不通过**、**未通过理由**、建议补做项；同一任务多次核查须**保留历次记录**（时间线或版本列表），不得覆盖删除历史判定。
- 用户界面侧：具备权限的用户须能从**任务详情**或**核查记录**入口查看上述历史；支持按项目、任务、时间、结论类型筛选（导出规则可选）。

**与测试 Agent 的关系**

- **测试 Agent** 负责设计/执行测试与缺陷闭环；其自身承担的任务在申报完成时，**同样**须经 **核查 Agent** 核验（例如测试报告完整性、覆盖度与需求对齐等）。
- **产品 Agent、项目经理 Agent** 所承担的任务（如文档定稿、里程碑达成、派工闭环等）申报完成时，**同样**必须经 **核查 Agent** 按约定标准核验。

**配置（概要）**

- 项目内须启用至少 **1 个核查 Agent 实例**（或与任务类型绑定的多实例策略由技术方案定义）；核查策略模板可按职能线区分（如后端偏契约与接口、前端偏交互与可访问性等），但**角色类型**统一为 **核查 Agent**。

## 5. 项目管理功能需求

### 5.1 项目入口与工作台

- 项目列表：名称、状态、参与 Agent 数量、最近活动、关联需求版本等。
- 项目详情：**真人成员**（系统用户）、**Agent 列表**（角色、主/子关系、当前任务、健康度/心跳可选）、**沟通与协作**、**任务看板或列表**、**需求文档**入口、**改进与变更记录**摘要；可从项目跳转至与本项目相关的**待办**（若有）。
- **项目成员角色**：**不**单独提供「仅负责拍板」的成员档位；用户在需求池中能否完成拍板，由平台 **RBAC**（如需求池决策类权限码，命名见技术方案）与 **项目内成员身份** 共同决定，避免为拍板单独维护一套成员类型。

### 5.2 任务与模块

- 功能模块与任务建议支持树或标签，与需求条目可**双向关联**。
- 任务状态须支持 **§4.5** 闭环，至少包含：**待派发**、**进行中**、**待核查**（执行方已申报完成，等待核查 Agent）、**核查中**、**已完成**（**仅**在核查 Agent **核查通过**后进入）、**阻塞**；**核查不通过**时任务须**退回进行中**并保留核查意见（**与历次核查记录一并持久化**，见 **§4.5**）。阻塞需记录原因与责任人 Agent。
- 原「待验收」语义由 **待核查 / 核查中** 承接；若需保留人工抽检节点，可作为可选状态叠加，**不得弱化**「非核查 Agent 通过不得标已完成」的硬规则。

### 5.3 改进记录（必录）

- 对每个 Agent：**配置变更**、**提示词/策略版本**、**人工反馈采纳**、**输出质量标注**等均写入**改进日志**。
- 日志需可检索、可按 Agent / 项目 / 时间过滤；支持导出（可选）。

### 5.4 定时任务

- 支持为项目或 Agent 配置**定时任务**：如每日站会摘要、进度巡检、需求冻结提醒、逾期任务升级给项目经理 Agent 等。
- 任务执行结果写入项目时间线或通知渠道（与平台通知体系对齐）。

### 5.5 运营大屏（实时动态）

- 提供**大屏视图**（支持常规大屏分辨率与全屏展示；可与普通项目看板共用数据源）。
- **实时动态**展示（刷新策略由技术方案约定，产品侧要求「观感实时」，如秒级～十秒级轮询或推送）：
  - 各 Agent **当前状态**：空闲 / 执行中 / 等待输入 / 阻塞 / 离线或异常（与健康检查对齐）；
  - **是否在干活**：与当前绑定任务、最近产出时间戳、心跳等关联展示；
  - **产出效率**：至少展示项目可理解的指标（如周期内完成任务数、交付物更新频次、可选 Token/调用成本等由技术方案选列），支持按 Agent、按职能线、按项目筛选。
- **告警**（大屏高亮 + 可选声音/订阅通知，具体渠道与平台通知对齐）：
  - **Agent 应干活但未干活**：例如已派工且超过阈值仍无进展、长时间无心跳且无合理解释；
  - **项目经理 Agent 无法协调**：在约定轮次或时限内仍未解除阻塞，或项目经理 Agent 明确标记「无法协调」时，**大屏告警**并保留事件编号便于追溯。

### 5.6 升级链路、项目经理协助记录与产品—需求池

**产品线 Agent 主导、人工拍板窄口径**

- 默认由 **产品 Agent（产品线智能体）** 完成：组织知识库检索、竞品/公开市场信息检索、需求表述修正建议、**多方案比选**与结论落地；**不应**将「每个需求决策」都升级为真人拍板。
- **仅当同时满足**下列条件的升级事项，才必须进入**需求池「待用户拍板」并等待真人用户**确认（缺一不可）：
  1. **组织知识库**中**无法检索到**可支撑该需求点的有效依据（或低于产品约定的置信度/覆盖率阈值，阈值在技术方案中配置并可在后台调整）；**且**
  2. **产品 Agent 在组织知识库与授权检索范围内执行检索后，仍无法得到**与争议点相关的**成熟产品**类有效命中（且无组织知识库直接依据），即**「搜不到成熟产品」**（判定见 **5.6.1**）。
- **若检索到多条竞品或多种市场做法**：产品 Agent **必须先做结构化对比**（对比维度须可配置，至少包含：目标用户与场景、功能边界、体验与风险、成本与实现复杂度、合规与安全等中的若干项），**择优选定一条**作为需求基线或推荐方案，并**强制记录对比过程与遴选理由**；该情形**由产品 Agent 闭环**，**不要求**真人拍板，**禁止**未经对比、无依据地**随意**选定其中一条。
- **若组织知识库中已有可采纳条目**，或 **产品 Agent 经检索能够发现**与争议点相关的**成熟产品**信息（判定见下节 **5.6.1**）：产品 Agent 应**直接**更新需求文档（或关联条目）并给出任务/派工调整建议，**无需**进入待用户拍板；需求池可记一条**已闭环**研判记录备查。

#### 5.6.1 以「Agent 检索能发现成熟产品」为闭环门槛

产品侧约定：**闭环的充分条件**是产品 Agent 使用**组织知识库**与/或**授权的竞品、公开市场等检索能力**完成查询后，在检索结果中能够识别出**至少一款已落地、面向真实用户可用的产品或服务**（商业产品、公开发布的 SaaS/App、稳定发行的开源产品/组件等）与当前争议需求点**存在可理解的关联**。**必须有可指名的「产品实体」**——即市场上**已经落地**、用户或客户**实际可获取或使用**的对象，**不能**仅靠抽象描述凑结论。

**须同时满足**

1. **已执行检索**：按规定走知识库与/或外部检索（渠道与范围由技术方案与合规策略约束），**留痕**检索要点（如关键词、数据源类型、检索时间可选）。
2. **命中对象为已落地产品**：检索结果中能提取出**具体产品名称或稳定标识**，且对应对象为**已发布、可获取、在用**的形态（正式发版产品、稳定版开源组件、广泛提供的平台能力等；排除明显个人 Demo、无实体的泛化空话——具体过滤规则可由技术方案配置）。
3. **与争议点可关联**：用一两句话说明该命中如何**支撑**当前需求判断（不要求长篇论证）。
4. **留痕（可审计）**：在「已闭环」记录中保存 **检索摘要 + 命中的成熟产品名称/标识 + 结论要点**；若有可保存的链接或摘录可一并附上，**不作为硬性前提**。

**下列情形一律不视为命中「成熟产品」（须走「多方案对比择优」或「待用户拍板」）**

- **仅有**学术论文、学术交流材料、纯理论或综述文档，**而无**可指名的、**已落地上市/公开发布可获取**的产品实体作为依托；
- **仅有**某概念的引用文档、白皮书、博客观点，**但说不清**是哪一款**真实在售/在用**的产品具备该能力；
- 检索结果中存在**多款成熟产品**且做法**明显不同**，却未按 5.6 节完成**结构化对比择优**；
- 检索**无法**得到任何可锚定到**具体已落地产品**的有效信息，或仅有与争议点**明显不对题**的噪音结果；
- 未实际执行检索或检索范围不符合组织策略（应由流程拦截，不视为有效闭环）。

*说明：「成熟 / 已落地」判定与检索白名单由技术方案配置；抽检规则另定。*

**升级顺序（执行链）**

1. **执行 Agent** → **项目经理 Agent**：说明无法执行原因；项目经理尝试协调。
2. **项目经理 Agent 仍无法解决** → **大屏告警**（与 5.5 联动），并**升级至产品 Agent**：研判是否为**需求不合理**、范围不清、验收标准缺失、与知识库/竞品实践冲突等。
3. **产品 Agent** 按上节规则执行：**知识库 → 检索成熟产品（5.6.1）→ 多方案则对比择优 → 写回结论**；**仅触发「待用户拍板」窄口径时**，才在需求池创建「待用户拍板」条目。
4. 各节点结论须**写回需求文档与/或需求池状态**（含已闭环记录），并驱动任务与派工**重排**（与第 6.2 节衔接）。

**项目经理协助记录（必录）**

- 须记录：**项目经理 Agent 解决了什么问题**（问题类型、根因摘要、采取动作）、**帮助了哪些 Agent**（Agent 标识、关联任务/线程）、时间、是否已解除阻塞。
- 记录须可在项目内检索，并纳入**审计与报表**（可选导出）。

**记录与可见性**

- 升级单、告警事件、需求池**待用户拍板**项、**已闭环**研判与竞品对比择优记录、用户拍板结果均须**持久化**且与项目/需求条目**关联**。
- 用户（有权限角色）在**需求池**中可查看**待用户拍板**列表、**已闭环**研判记录及历史已确认项。

#### 5.6.2 真人用户（系统用户）与待办模块

- **真人项目成员**与**待拍板责任人**均对应平台 **系统用户**：账号体系、登录态与角色权限与 **[权限产品线需求](权限产品线需求.md)** 中的用户/角色/RBAC **一致**，不在本产品线重复造用户模块。**不**引入「仅拍板」专用项目成员角色；谁具备拍板操作权，由 **RBAC + 项目成员** 组合判定（与技术方案 §0 对齐）。
- 用户 **登录后**，须能处理与本账号相关的 **待用户拍板**：提供 **待办** 模块 **与/或** **需求池** 中的可操作入口（至少一种清晰可达，可**双向跳转**；具体信息架构由技术方案定稿）；仅对**具备该项目权限或拍板指派关系**的用户展示对应条目。
- **可选聚合**：是否将 **告警待确认、定时任务失败** 等与拍板**合并进同一「待办」列表**为**可选**能力，**不强制**首期实现；**必选**仅为 **待用户拍板** 的可见、可处理与状态闭环（无论主要入口标为「待办」还是「需求池」子区，产品验收以权限用户能完成拍板为准）。

### 5.7 交付发版与发布版本归档

- 当项目达到**可对外发版/上线**里程碑时，支持发起**发版归档**操作（可与人工审批或发布 checklist 结合，流程细节见技术方案）。
- 每次发版须生成一条**发布版本记录**（版本号、发布时间、关联代码/制品或发布说明的引用字段由技术方案定义），并将该时间点前后的项目态做**快照级归档**（至少包含：需求基线版本、模块清单、主要任务与结论状态）。
- 归档内容须显式包含可供后续项目借鉴的**过程与经验**，包括但不限于：
  - **方案/实现对比过程**（若存在多方案比选、评审结论）；
  - **Agent 协调过程中出现的典型问题**、根因分类、解决路径；
  - **协调停滞**的案例：**为何停滞**（依赖、需求、环境、资源、模型能力边界等）、如何解除或如何规避；
  - **项目经理协助记录**、升级链路中的**高价值结论**（脱敏后纳入经验摘要的可选项）；
  - 可复用的**功能模块标签/边界说明**（便于与后续项目做**模块级匹配**）。
- 已归档的发布版本**只读**；修正历史认知应通过**新发布版本**或**勘误附录**追加，不覆盖篡改原归档快照（与审计一致）。

### 5.8 项目隔离与跨项目经验共享

**隔离（硬约束）**

- **项目维度强隔离**：不同项目的任务、需求草稿、配置、**Agent 会话与上下文存储**、运行中协作线程**不得跨项目混用或默认互通**；任一项目的操作**不得写入**其他项目的业务数据。
- **新项目的 Agent 会话**一律归属**新项目空间**；历史项目的实时会话、令牌上下文、未归档草稿**不可**被新项目 Agent 直接接续。

**共享（受控、只读或可复制的知识层）**

**组织知识库的来源（产品约定）**

- **来源一 — 历史项目优质经验**：上一项目或既往项目在**完成交付、发版归档**时，将评审认定为**质量较好、可复用**的经验（协调教训、模块蓝图、方案对比结论、排障总结等）**回流**至组织知识库，供后续项目与产品 Agent 检索。
- **来源二 — 人工录入**：具备权限的用户通过管理界面**手动新增、编辑、分类、下架**知识库条目，补充归档尚未覆盖的领域或纠偏。
- 上述两类共同构成与 **2.1「组织知识库」** 模块一致的**组织级可检索内容**；**发版归档**产生的快照与提炼材料（摘要、模块蓝图、协调 playbook、常见问题等）经回流后进入来源一，与具体项目运行态分离，技术实现上与「组织经验与模块复用」数据流对齐，避免重复维护多份矛盾口径。
- 新建项目时，可授权新项目**检索与订阅**组织知识库及归档提炼物：当新项目的**功能模块**与历史归档中的模块在标签/结构上匹配时，支持**一键引用**（只读参照）或**复制为本项目模块蓝图/任务模板**（复制后属于新项目资产，修改不影响源归档）。
- **新项目 Agent 可使用**被授权的组织经验作为**外部知识注入**（如检索增强、系统提示中的「组织最佳实践」区块），但**不得**将其他项目的**私有会话原文**默认注入；涉及敏感内容时遵循脱敏与权限策略（见第 7 节）。
- 产品需保证：共享的是**经验与可复用结构**，而非**跨项目实时状态耦合**，避免项目间相互影响。

## 6. 需求文档与变更驱动

### 6.1 需求文档界面

- 提供**专用功能界面**：支持富文本或结构化段落（史诗 / 特性 / 用户故事）；版本保存与对比（至少保留历史版本列表）。
- 用户（含产品角色）可**在线编辑、批注、改进**；支持协作编辑冲突策略（首期可「保存即新版本」简化）。

### 6.2 变更影响与重排

- 当需求发生**实质性变更**时，系统应支持 **Agent 主导的分析与建议 + 可追溯留痕**；**真人用户确认**仅在第 **5.6** 节「待用户拍板」窄口径触发时强制要求，其余环节不默认捆绑人工确认。
- 产品侧应支持：
  - 标识**受影响的功能点 / 模块 / 任务**；
  - 触发**项目经理 Agent** 组织各职能主 Agent **重新评估工时与依赖**；
  - 对未开始或进行中的任务生成**调整建议**（取消、拆分、合并、重排优先级）。
- 目标：需求文档为驱动源之一时，**全项目相关交付与 Agent 分工**能随文档演进**可追踪地调整**，避免静默漂移。

### 6.3 需求池（待用户拍板、已闭环与竞品参照）

- **需求池**与需求文档条目、用户故事或功能点**可关联**；状态至少区分：**待用户拍板**（仅 5.6 窄口径）、**已闭环（产品 Agent）**、用户已确认合理、用户已确认需变更、已关闭等（具体状态集可迭代）。
- **待用户拍板**卡片：仅在「知识库无有效依据 **且** **检索后仍搜不到**符合 **5.6.1** 的**已落地成熟产品**」时由系统生成。已登录用户在 **待办** 与/或 **需求池** 中处理拍板项（见 **§5.6.2**）；可在**需求池**中查看全量列表与**已闭环**研判记录。
- **已闭环（产品 Agent）卡片：用于记录知识库命中、检索命中成熟产品（5.6.1）或多竞品对比择优**后的结论与遴选依据，**不要求**用户确认，但必须**可审计**；闭环节点**至少**保存 **检索摘要 + 命中的成熟产品标识 + 结论要点**。
- 产品 Agent 在池内维护字段示例：**问题描述**、知识库检索摘要、**竞品对比表**（多方案时必填）、**选定方案与理由**、**待用户决策点**（仅待拍板项非空）；用户拍板后**留痕**并触发文档与任务更新。

## 7. 权限与安全（摘要）

- **系统用户与拍板入口**：真人能力依赖平台统一 **用户与权限** 体系；在 **待办** 与/或 **需求池** 中展示的拍板项须受 **RBAC** 与**项目成员/指派关系**双重约束（无权限者不可见、不可操作）。**不**为拍板单独增设「仅负责拍板」的项目成员类型。
- 项目级数据隔离：用户仅能访问被授权的项目与 Agent 配置；**项目间默认零互通**（与第 5.8 节一致）。
- **组织级归档与经验库**：按角色控制「可读 / 可检索 / 可引用 / 可复制蓝图」；**组织知识库**的**人工录入、编辑、下架**须单独授权；跨项目引用须**显式授权或组织策略**允许，并记录审计。
- 敏感沟通内容可按角色脱敏；导出与 API 需鉴权；注入新项目 Agent 的**历史经验**须经过**脱敏与范围裁剪**策略。
- Agent 调用外部工具与数据的范围需在配置中显式声明并接受审计。

## 9. 验收要点（摘要）

- 可创建项目并**新增多类 Agent**（含主/子关系），在项目内看到列表与状态。
- 可观察到 **Agent 间沟通记录**（模拟或真实编排均可，以产品演示路径为准）。
- 子 Agent 完成工作后能触发 **「要活」流程**，主 Agent 向项目经理 Agent **请求派工**，项目经理按 **模块与进度** 更新任务分配。
- 任意职能 Agent 申报任务完成后，任务进入 **待核查/核查中**，由 **核查 Agent** 检索与验证；**核查通过**后才可进入 **已完成**；**测试/产品/项目经理** 职能申报完成同样适用；**不得以执行方自述代替核查结论**。**每一次核查通过/不通过**均在系统内**可查、可追溯**（持久化记录，含时间与依据摘要）。
- 每个 Agent 的**改进记录**可查询；**定时任务**可按配置执行并在界面留痕。
- 需求文档可编辑并产生新版本；发起变更后，能展示**受影响功能点**并驱动任务/分工的**重排流程**（Agent 主导建议与批量状态更新可留痕；**仅 5.6 窄口径**须**用户拍板**后落库）。
- **运营大屏**可实时或近实时展示 Agent **是否在干活**与**产出效率**；Agent 长时间不干活或项目经理**无法协调**时出现**可见告警**。
- 模拟或真实编排下：派工后无法执行的 Agent 能**主动向项目经理说明原因**；系统中可查**项目经理协助了哪些问题、哪些 Agent**；升级至产品后，**检索能命中已落地成熟产品**（非纯文档/学术）时**已闭环**记录符合 **5.6.1**；**多竞品场景**可见**对比与择优留痕**；**仅窄口径**下产生**待用户拍板**，**已登录用户**在 **待办** 与/或 **需求池** 可完成处理；**待办聚合多类事项为可选**（§5.6.2）。
- **组织知识库**可见来自**归档回流**与**人工录入**的条目；产品 Agent 可检索命中。
- 项目发版后可生成**发布版本归档**，其中包含**协调问题、停滞原因与经验摘要**等；**不同项目会话与数据互不影响**；新项目可**引用/复制**历史归档中的**共性模块蓝图**，且新项目 Agent 在**本会话空间**内可使用**被授权的组织经验**辅助工作。
