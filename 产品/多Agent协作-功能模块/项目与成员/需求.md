# 项目与成员视图（节选）

> 完整版：[多Agent协作与项目管理产品线需求.md](../../多Agent协作与项目管理产品线需求.md)

## §2.1 模块说明

| 模块 | 说明 |
|------|------|
| 项目与成员视图 | 项目 CRUD；**真人成员**（系统用户，依赖平台用户与权限模块，与项目绑定）；**不设**「仅负责需求池拍板」的独立成员类型，拍板能力由 **RBAC + 项目成员角色**（如 OWNER / MEMBER / VIEWER，枚举与技术方案一致）组合授予；项目内 Agent 列表、状态、所属职能线 |

---

## §5.1 项目入口与工作台

- 项目列表：名称、状态、参与 Agent 数量、最近活动、关联需求版本等。
- 项目详情：**真人成员**（系统用户）、**Agent 列表**（角色、主/子关系、当前任务、健康度/心跳可选）、**沟通与协作**、**任务看板或列表**、**需求文档**入口、**改进与变更记录**摘要；可从项目跳转至与本项目相关的**待办**（若有）。
- **项目成员角色**：**不**单独提供「仅负责拍板」的成员档位；用户在需求池中能否完成拍板，由平台 **RBAC**（如需求池决策类权限码，命名见技术方案）与 **项目内成员身份** 共同决定，避免为拍板单独维护一套成员类型。

---

## §5.6.2 中与「真人项目成员」相关的表述（交叉引用）

- **真人项目成员**与**待拍板责任人**均对应平台 **系统用户**；**不**引入「仅拍板」专用项目成员角色；谁具备拍板操作权，由 **RBAC + 项目成员** 组合判定。

完整待办与拍板入口见 [用户待办](../用户待办/需求.md)。

---

## §7 中与项目访问相关的摘要（交叉引用）

- 项目级数据隔离：用户仅能访问被授权的项目与 Agent 配置；**项目间默认零互通**（与 §5.8 一致）。

---

## §9 验收要点（与本模块相关）

- 可创建项目并新增多类 Agent，在项目内看到列表与状态（与 Agent 管理协同验收）。
