# 权限系统技术方案

本文档与项目技术栈对齐：**Java 24、Spring Boot 3.5、MyBatis、MySQL、Redis、MinIO**；前端 **Vue 3 + Vite + Pinia**。MinIO 与权限无强耦合，仅在与「用户头像/附件」等业务结合时使用。

---

## 1. 总体架构

```
┌─────────────┐     HTTPS      ┌──────────────────┐
│  Vue3+Vite  │ ◄────────────► │  Spring Boot API │
│  Pinia/Router│                │  Spring Security │
└─────────────┘                └────────┬─────────┘
                                        │
                         ┌──────────────┼──────────────┐
                         ▼              ▼              ▼
                     MySQL          Redis         (MinIO)
                   用户/角色/权限    会话/缓存/黑名单   非核心
```

**原则**

- **鉴权以服务端为准**：前端路由与按钮仅提升体验，不可作为安全边界。
- **权限标识统一**：数据库中 `perm_code` 与后端 `@PreAuthorize`（或自定义元数据）、前端 `v-permission` 使用同一字符串或约定映射表。

---

## 2. 认证方案（推荐）

### 2.1 JWT + Redis（推荐于分布式）

| 环节 | 说明 |
|------|------|
| 登录 | 校验用户名密码 → 生成 JWT（含 `sub` 用户 ID、`jti`、过期时间） |
| 存储 | Access Token 短期（如 15m～2h）；可选 Refresh Token 长期，存 Redis 或 DB |
| 登出 | 将 `jti` 写入 Redis 黑名单，校验时拒绝；或删 Refresh 记录 |
| 校验 | `OncePerRequestFilter` 解析 JWT，加载用户与权限到 `SecurityContext` |

*若首期单机且团队更熟悉 Session，可采用 **Spring Session + Redis**，方案等价替换令牌层即可。*

### 2.2 密码存储

- 使用 **BCrypt**（Spring Security Crypto）对密码哈希，禁止明文或可逆加密。

## 3. 授权模型

### 3.1 RBAC

采用 **用户 — 角色 — 权限（多对多）**，菜单作为带 `perm_code` 与 `route_meta` 的树形资源，可与权限表合并或关联。

- 用户可绑定多个角色；最终权限 = 角色权限并集。
- **接口级权限**：每个 API 定义 `perm_code`（如 `system:user:add`），与角色绑定。
- **菜单级**：`menu` 记录关联 `perm_code`，前端根据返回树过滤；路由 `meta.permission` 与之一致。

### 3.2 Spring Security 集成要点（Spring Boot 3 / Java 24）

- 依赖：`spring-boot-starter-security`（版本与 Spring Boot BOM 一致）。
- 配置类：以 **`SecurityFilterChain` Bean** 为主；`authorizeHttpRequests`、`AbstractHttpConfigurer::disable` 等与 Spring Security 6 一致。
- 方法级：`@EnableMethodSecurity(prePostEnabled = true)` + `@PreAuthorize("hasAuthority('system:user:add')")`，或自定义 `PermissionEvaluator` 从 DB/缓存读权限集合。
- 白名单：`/api/auth/login`、`/actuator/health`（按需）等 `permitAll`。

## 4. 数据模型（MySQL + MyBatis）

### 4.1 核心表（建议）

| 表名 | 用途 |
|------|------|
| `sys_user` | 用户：用户名、密码哈希、昵称、状态、部门 ID（可选）等 |
| `sys_role` | 角色：编码、名称、备注 |
| `sys_permission` | 权限点：编码、名称、类型（菜单/按钮/API）、父 ID、排序、路由 path、component |
| `sys_user_role` | 用户-角色 |
| `sys_role_permission` | 角色-权限 |

**字段建议**

- `sys_user.status`：`0` 禁用 `1` 正常；登录时拒绝禁用用户。
- `sys_permission.perm_code`：全局唯一，供前后端与注解使用。
- 菜单树：`parent_id` + `sort`；`type` 区分目录/菜单/按钮/API。

### 4.2 MyBatis

- Mapper XML 或注解实现用户按用户名查询、角色 ID 列表、权限编码列表。
- 登录成功后一次查询组装「权限编码集合 + 菜单树」，减少 N+1（可用多表关联或分步查询 + 应用层组装）。

## 5. Redis 使用约定

| Key 模式 | 用途 | TTL |
|----------|------|-----|
| `auth:token:blacklist:{jti}` | 登出黑名单 | 与 token 剩余寿命对齐 |
| `auth:user:perms:{userId}` | 用户权限编码缓存 | 5～30 分钟，变更时删除 |
| `auth:user:menus:{userId}` | 菜单树缓存（可选） | 同上 |
| `captcha:{uuid}` | 图形验证码 | 1～5 分钟 |

**缓存失效**：在用户-角色、角色-权限变更时，删除对应 `userId` 的 perm/menu 缓存；或引入 `perm_version` 存在 JWT 自定义 claim 中与服务端版本比对（复杂度较高，二期考虑）。

## 6. 接口约定（示例）

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/auth/login` |  body：username、password → token + 基础用户信息 |
| POST | `/api/auth/logout` |  注销，写黑名单 |
| GET | `/api/auth/me` |  当前用户、权限列表、菜单树 |
| CRUD | `/api/system/users` |  用户管理，需 `system:user:*` |
| CRUD | `/api/system/roles` |  角色管理 |
| CRUD | `/api/system/menus` |  菜单/权限树维护 |

统一响应体建议：`{ code, message, data }`；`401/403` 与业务码文档对齐。

## 7. 前端实现（Vue 3 + Vite + Pinia）

### 7.1 Pinia Store

- `useUserStore`：`token`、`userInfo`、`perms`（`Set` 或数组）、`menus`。
- 登录后调用 `/api/auth/me` 或通过 login 响应一次性写入；**动态路由**：根据 `menus` 生成 `router.addRoute`。
- Axios 拦截器：附加 `Authorization: Bearer xxx`；`401` 清 token 跳转登录；`403` 可全局提示。

### 7.2 路由守卫

- `beforeEach`：无 token → 登录页；有 token 无动态路由 → 拉取菜单并注册后 `next(to.fullPath)`。
- 路由 `meta.permission`：若当前路由需要权限且用户无 `perm`，跳转 403。

### 7.3 指令 / 组件

- 自定义 `v-permission="'system:user:add'"`：无权限不渲染按钮。

## 8. 安全与运维

- 生产环境强制 HTTPS；Cookie 若使用须 `HttpOnly`、`Secure`（Session 方案）。
- JWT **密钥**放配置中心或环境变量，禁止入库或提交仓库。
- 管理接口限流（可选）：Redis + 注解，防止爆破与滥用。
- 操作审计表 `sys_oper_log`：记录模块、操作人、IP、参数摘要、结果（与产品需求联动）。

## 9. 与产品需求的对应关系

| 产品需求 | 技术落点 |
|----------|----------|
| 登录/登出 | JWT + Redis 黑名单或 Session |
| 用户/角色/菜单管理 | MySQL 表 + MyBatis + 管理 API |
| 前端菜单与按钮控制 | Pinia + 动态路由 + `v-permission` |
| 接口鉴权 | Spring Security + `@PreAuthorize` / 自定义 Filter |
| 权限变更生效 | 缓存删除 + 可选短 TTL；前端可提示重新登录 |
