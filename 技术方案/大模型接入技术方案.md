# 大模型接入 — 技术方案

## 1. 架构概览

```
前端 (Vue)  →  POST /api/llm/chat  →  LlmChatRouter
                                         ↓
                    ┌────────────────────┼────────────────────┐
                    ▼                    ▼                    ▼
           OpenAiCompatible      AnthropicMessages     DashScopeCompat
              ChatClient            ChatClient            ChatClient
                    ↓                    ↓                    ↓
               厂商 HTTPS              厂商 HTTPS            厂商 HTTPS
```

- **通道表**持久化：`provider_code`、`base_url`、`api_key`（密文）、`default_model` 等。
- **LlmChatRouter** 按 `provider_code` 与 `protocol` 选择客户端；同一协议共用实现。

---

## 2. 协议与适配器

| 协议枚举 | 适用厂商（示例） | 实现说明 |
|----------|------------------|----------|
| `OPENAI_COMPAT` | OpenAI、DeepSeek、MiniMax（兼容）、GLM（兼容）、Moonshot、Groq、Azure、Ollama、多数自建网关 | `POST {base}/v1/chat/completions`，Bearer Key |
| `ANTHROPIC` | Claude | `POST {base}/v1/messages`，`x-api-key` + `anthropic-version` |
| `DASHSCOPE_COMPAT` | 阿里云通义兼容 OpenAI 模式 | 与 OpenAI 兼容路径一致，Base 为兼容地址 |
| `GEMINI` | Google Gemini | 后续：GenerativeLanguage REST |

首期代码：**完整实现 OPENAI_COMPAT**；**ANTHROPIC** 与 **DASHSCOPE_COMPAT** 提供可用 HTTP 调用骨架（字段与路径以官方文档为准，可在配置中调整 `base_url`）。

## 3. 数据模型（MySQL）

### 3.1 `llm_channel`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | |
| owner_user_id | BIGINT | 归属用户 |
| provider_code | VARCHAR(32) | 内置字典编码，如 `deepseek`、`openai` |
| channel_name | VARCHAR(64) | 展示名 |
| base_url | VARCHAR(512) | 无尾斜杠，如 `https://api.deepseek.com` |
| api_key_cipher | TEXT | AES 密文 Base64 |
| default_model | VARCHAR(128) | 可空 |
| protocol | VARCHAR(32) | 冗余存储，便于路由 |
| extra_json | VARCHAR(1024) | 扩展 |
| status | TINYINT | 1 启用 0 禁用 |
| created_at / updated_at | DATETIME | |

### 3.2 密钥加密

- 配置项 `aiburst.llm.crypto.secret`：32 字节 ASCII（或后续改为 KMS）。
- 应用内 **AES-256** 加解密，接口响应**不回传** Key。

## 4. API 约定

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/llm/providers` | 内置厂商/协议字典 |
| GET | `/api/llm/channels` | 当前用户通道列表（脱敏） |
| POST | `/api/llm/channels` | 新建 |
| PUT | `/api/llm/channels` | 更新 |
| DELETE | `/api/llm/channels/{id}` | 删除 |
| POST | `/api/llm/chat` | 同步对话；`messages` 为完整多轮列表（user/assistant 交替，以各协议要求为准） |

统一响应包：`ApiResult`；错误信息避免泄漏 Key。二期可增设 **SSE 流式** 端点（或同路径 `Accept: text/event-stream`），与首期 JSON 并存。

---

## 5. 安全

- 所有 `/api/llm/**`（除未来可能的内部 health）需 **登录 + 权限**。
- `chat` 校验 `channel.owner_user_id == currentUserId`（管理员突破规则可后续加角色）。
- 日志：仅记录 channelId、provider、model、耗时与错误码。

---

## 6. 前端

- **通道配置页**：表格 + 表单；下拉选择 `provider_code`；Base URL 预填可编辑；Key 密码框；保存后列表不显示 Key。
- **对话体验页（`Playground.vue`）**
  - **布局**：顶部工具条（通道、可选模型覆盖、清空会话）；中部**可滚动会话区**（用户 / 助手气泡左右分区、时间戳）；底部多行输入 + 发送（`llm:chat:invoke`）。
  - **数据流**：会话状态维护在浏览器内存中的消息列表；每次发送对 `POST /api/llm/chat` 传入**当前列表全文**作为 `messages`，实现多轮上下文；接口仍为**同步**返回，收到后追加助手气泡。
  - **体验**：请求中展示生成中占位、防重复提交、新消息**滚动到底**；`usage` 置于折叠面板，避免干扰阅读。
  - **流式**：首期不实现；与产品线「二期 SSE」对齐，届时可改为 `EventSource` 或 fetch ReadableStream 消费增量内容。
