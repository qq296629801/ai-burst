# AI Burst 项目开发规范

本文档在体系上对齐**《阿里巴巴 Java 开发手册》**（及社区版 P3C 等同类实践）的划分方式，结合本仓库技术栈（**Java 8、Spring Boot 2.7、MyBatis、Redis、Vue 3 + Vite + Pinia**）做落地约定。规约优先级分为：

- **【强制】**：评审与 CI 可据此卡关；违反需说明理由并记录例外。
- **【推荐】**：默认遵守，特殊情况可偏离并注释说明。
- **【参考】**：团队共识基础上选用。

---

## 1. 编程规约（Java）

### 1.1 命名风格

| 类型 | 【强制】 | 说明 |
|------|----------|------|
| 类名 | `UpperCamelCase` | 如 `LlmChannelService`，避免缩写歧义。 |
| 方法/参数/成员 | `lowerCamelCase` | 如 `selectByOwner`、`ownerUserId`。 |
| 常量 | `UPPER_SNAKE_CASE` | 如 `TOKEN_KEY`、`PREFIX`。 |
| 包名 | 全小写、单数语义 | 本仓库：`com.aiburst`、`com.aiburst.llm` 等。 |

**【推荐】** 业务含义优先于缩写；`DTO/VO` 等后缀与现有代码保持一致。

**【参考】** 与《手册》一致：抽象类可用 `Abstract`/`Base` 前缀，异常类以 `Exception` 结尾，测试类以被测类名 + `Test` 命名。

### 1.2 常量与魔法值

**【强制】** 禁止在业务代码中直接使用无含义魔法字符串/数字（错误码、权限标识、Redis Key 前缀等应集中为常量或配置）。本仓库示例：`com.aiburst.common.constants.AuthRedisKeys`（`auth:token:blacklist:`、`auth:user:perms:` 等）；前端本地存储键名见 `src/constants/storage.js`。

**【推荐】** 配置项使用 `application.yml` + `@ConfigurationProperties` 或 `@Value`，密钥类配置走环境变量，不入库、不进 Git。

### 1.3 格式与风格

**【强制】** 与团队 IDE 统一：UTF-8、4 空格缩进（与现有工程一致）、单行不宜过长；必要处换行保持可读。仓库根目录 **`.editorconfig`** 已约定 Java 4 空格、前端 2 空格、换行符 LF。

**【推荐】** 提交前执行格式化；Java 可使用 Checkstyle / P3C 插件辅助。

### 1.4 OOP 规约

**【强制】**

- 对外 API（Controller）入参使用 **DTO** 并配合 **Bean Validation**（`@Valid`、`@NotNull` 等），避免直接暴露持久化实体。
- **equals / hashCode**：若重写需保证契约；实体作集合键需谨慎。
- 禁止在子类中隐藏父类同名 public 方法语义不清的重写（除非有意设计）。

**【推荐】**

- 优先组合与清晰分层，避免过深继承。
- 工具类构造器私有化（如 `UrlJoinUtil`）。

### 1.5 集合与并发

**【强制】** 判断集合空：使用 `CollectionUtils` / `isEmpty()`，不要仅依赖 `size() == 0`（部分集合 `size` 成本高）。

**【推荐】** 并发场景对共享状态加锁或使用线程安全结构；本项目中 **JWT + 无状态** 为主，避免在静态域缓存用户态数据。

### 1.6 控制语句与注释

**【强制】** 少用无意义 `else` 嵌套，优先早返回；`switch` 必须有 `default`（若语言版本支持）。

**【推荐】** 仅对非显而易见的业务规则、安全边界、协议差异写简短注释；避免注释复述代码。

## 2. 异常与日志

### 2.1 异常处理

**【强制】**

- **禁止** 捕获 `Exception` 后空吞或仅 `printStackTrace`；必须记录或向上抛出并转化为统一响应。
- 业务可预期错误使用 **明确异常或 `IllegalArgumentException` + 全局处理**（如参数校验）；对外 HTTP 状态与 `ApiResult` 语义一致。
- 调用外部大模型等上游失败：使用专用异常（如 `LlmInvocationException`），**不在日志中打印 API Key**。

**【推荐】** 错误信息对用户可读、对开发可定位；敏感信息脱敏。

### 2.2 日志规约

**【强制】** 使用 SLF4J：`log.info/warn/error`，占位符 `{}`，**禁止**字符串拼接大对象。

**【推荐】** 级别：ERROR（需关注）、WARN（可恢复/降级）、INFO（关键链路）、DEBUG（开发/排障）。生产默认 INFO；包级 `com.aiburst.mapper` 的 DEBUG 仅开发环境开启。

## 3. 安全规约

**【强制】**

- 鉴权以 **服务端** 为准；权限标识与数据库、`@PreAuthorize`、前端 `v-permission` 一致。
- 密码 **BCrypt**；通道 **API Key** 密文存储；JWT 密钥、加密盐禁止提交仓库默认值用于生产。
- 所有除登录外的业务接口需认证；CORS 白名单明确配置。

**【推荐】** 依赖漏洞定期扫描（Maven/npm audit）；HTTPS 部署。

## 4. MySQL 与 MyBatis

### 4.1 建表与字段

**【强制】**

- 表名、字段名 **小写 + 下划线**；必备：`id`、业务所需审计字段（如 `created_at`/`updated_at`）。
- 字符集 **utf8mb4**；存储引擎 **InnoDB**。
- 布尔/状态类使用 **TINYINT** 并注释取值含义。

#### 4.1.1 表名前缀与一致性（本仓库约定）

| 前缀 | 用途 | 示例 |
|------|------|------|
| `sys_` | 平台账号、RBAC、菜单权限 | `sys_user`、`sys_role`、`sys_permission`、`sys_user_role`、`sys_role_permission` |
| `llm_` | 大模型通道等业务域表 | `llm_channel` |

**【强制】** 禁止表名、字段名使用驼峰、大写或混合大小写（如 `SysUser`、`userName`）；禁止无域前缀的泛化表名导致冲突（新表按域选 `sys_` / `llm_` / 未来可扩展 `biz_` 等，并在评审中说明）。

**【推荐】** 二级索引命名：`idx_<表名简写>_<字段>`，全小写，如 `idx_llm_channel_owner`。

#### 4.1.2 SQL 脚本书写

**【推荐】** Flyway 脚本中 SQL **关键字大写**（`CREATE TABLE`、`INSERT INTO`），**标识符（表/列）小写**，与现有 `V1__init.sql`、`V2__llm_module.sql` 保持一致。

**【推荐】** 单表索引不宜过多；区分度高的列优先；联合索引注意最左前缀。

### 4.2 SQL 与 ORM

**【强制】**

- **禁止** 拼接字符串方式组织 SQL（防注入）；使用 `#{}` 预编译。
- `SELECT` 必须指定列，避免 `SELECT *`（除非有充分理由且团队认可）；Mapper XML 使用 `<sql id="Xxx_Column_List">` 统一列清单并与 `resultMap` 对齐。
- 变更结构通过 **Flyway** 版本脚本，禁止手工改生产库无记录。

**【推荐】** XML Mapper 与接口同业务域分包；复杂 SQL 写注释说明业务含义。

## 5. 工程结构与应用分层

**【强制】** 保持与现有分层一致：

| 层级 | 职责 |
|------|------|
| `controller` | 参数校验、鉴权注解、组装 `ApiResult`，不写复杂业务。 |
| `service` | 业务编排、事务边界（`@Transactional`）。 |
| `mapper` + XML | 数据访问，无业务分支。 |
| `dto` / `entity` | 入参出参与持久化模型分离。 |
| `config` / `security` | 横切配置与安全。 |

**【推荐】** 新业务子域可参考 `com.aiburst.llm`：**独立包 + 独立 Flyway 脚本前缀**，避免与 `sys_*` 强耦合。

**【参考】** 《手册》二方库/三方库：优先 Spring BOM 管理版本；新增依赖需说明用途与安全许可。

## 6. 接口与统一响应

**【强制】**

- 统一 `ApiResult`：`code`、`message`、`data`；HTTP 状态与业务错误语义一致（如 401/403 与全局处理器）。
- REST 资源路径风格与现有 `/api/auth/*`、`/api/system/*`、`/api/llm/*` 保持一致。

**【推荐】** 分页结构统一（如 `PageResult`）；列表接口文档化筛选参数。

---

## 7. 前端（Vue 3）规约

**【强制】**

- 组合式 API / `<script setup>` 与现有页面风格一致；路由与权限依赖后端菜单与 `v-permission`。
- HTTP 经统一封装（`src/api/http.js`），不在业务组件内散落 `axios` 实例配置。

**【推荐】**

- 组件与视图命名清晰；大表单拆子组件。
- 环境相关地址走 Vite 代理或环境变量，禁止硬编码生产域名。

**【参考】** ESLint + Prettier 与仓库配置对齐；提交前 `npm run build` 可通过。

---

## 8. 单元测试与质量

**【推荐】** 核心领域逻辑、加密/权限工具类补充单元测试；关键接口可预留集成测试（Testcontainers 可选）。

**【强制】** 合并主分支前：**后端 `mvn -DskipTests` 或含测试通过**、**前端 `npm run build` 通过**（以团队 CI 为准）。

---

## 9. Git 与协作

**【强制】**

- 提交信息语义清晰（中文或英文完整句均可，团队统一即可）；禁止大块无关格式化与业务混在同一提交。
- 密钥、本机路径、内网地址不得提交。

**【推荐】** 功能分支开发 + Merge/MR 评审；与需求/缺陷单关联。

## 10. 与本项目相关的专项约定

| 项 | 约定 |
|----|------|
| 权限缓存 | 变更用户/角色/权限后注意 Redis 缓存失效策略（见权限技术方案）。 |
| 大模型通道 | 仅 owner 可操作自己的 `llm_channel`；上游错误映射为 `LlmInvocationException`，不泄漏 Key。 |
| Flyway | 新脚本版本号递增；禁止修改已发布环境的已执行脚本内容。 |
